Rust中的错误主要分为两大类：

  • 可恢复错误： 通常用于从系统全局角度来看可以接受的错误，例如处理用户的访问、操作等错误，这些错误只会影响某个用户自身的操作进程，而不会影响到其他用户，也不会影响到系统的正常运行，因此可以使用Result类型来处理；
  • 不可恢复错误： 通常用于从系统全局角度来看不能接受的错误，例如文件不存在、网络连接失败等错误，这些错误会影响整个系统的正常运行，因此需要使用panic来处理；

backtrace 栈展开

栈展开(也称栈回溯)，它包含了函数调用的顺序，当然按照逆序排列:最近调用的函数排在列表的最上方。

Rust 的 backtrace 跟其他语言中的一样:阅读 backtrace 的关键是从头开始读直到发现你编写的文件。这就是问题的发源地。这一行往上是你的代码所调用的代码;往下则是调用你的代码的代码。

要获取到栈回溯信息，你还需要开启 debug 标志，当不使用 –release 参数运行 cargok用。同时，栈展开信息在不同操作系统或者 Rust 版本上也有所不同。

主动触发

Rust 为我们提供了 panic! 宏，当调用执行该宏时，程序会打印出一个错误信息，展开报错点往前的函数调用堆栈，最后退出程序

panic 时的两种终止方式

  • 栈展开(unwinding)(默认): Rust 会回溯栈上数据和函数调用，因此也意味着更多的善后工作，好处是可以给出充分的报错信息和栈调用信息，便于事后的问题复盘

  • 直接终止(abort): 不清理数据就直接退出程序，善后工作交与操作系统来负责

对于绝大多数用户，使用默认选择是最好的，但是当你关心最终编译出的二进制可执行文件大小时，那么可以尝试去使用直接终止的方式，例如下面的配置修改cargo.toml 文件，实现在 release 模式下遇到 panic 直接终止:

Result

大部分错误并没有严重到需要程序完全停止执行。有时候，一个函数失败，仅仅就是因为一个容易理解和响应的原因。例如，如果因为打开一个并不存在的文件而失败，此时我们可能想要创建这个文件，而不是终止进程。

  • 泛型参数 T 代表成功时存入的正确值的类型，存放方式是 Ok(T)；
  • E代表错误时存入的错误值，存放方式是 Err(E)；
